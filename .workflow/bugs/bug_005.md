## [BUG-005] Page Persistence Restore Fires Infinitely, Never Jumps to Saved Page

- **Status**: Resolved
- **Discovered**: 2026-04-02
- **Fixed**: 2026-04-02
- **File**: `lib/ui/screens/reader/reader_screen.dart`

### Symptoms
- Opening a book with a non-zero `lastReadPage` did not restore scroll position.
- The reader always started at page 1 regardless of what was saved in the DB.
- Additionally, scrolling was erratic — the view would jump back unexpectedly every ~2 seconds.

### Root Cause

Two related bugs in `_restoreToPage` and `_scheduleRestore`:

**Bug A — `_didInitialRestore` never set on failed lookup:**
`_restoreToPage` sets `_didInitialRestore = true` only *after* `Scrollable.ensureVisible` completes. If the saved page number wasn't found in any block's `sourcePages` (e.g. page `1` exists in DB but the OCR block starts at page `1` of a different numbering), the function returned early at `index == -1` without ever marking the restore as done. The 2s periodic `readerRefreshProvider` timer caused `build()` to re-call `_scheduleRestore` every cycle, which continuously re-queued `_restoreToPage` → index not found → repeat forever.

**Bug B — `_toggleAiMode` set `_pendingRestorePage` but fix made `_didInitialRestore` permanent:**
After fixing Bug A (moving the flag set to top-of-function), the AI mode toggle which sets `_pendingRestorePage` to scroll back to position would silently skip the restore because `_didInitialRestore` was already `true` and never reset.

### Fix

In `_restoreToPage`, moved both `_didInitialRestore = true` and `_pendingRestorePage = null` to the **top** of the function, before any early-return logic. This ensures the restore is only ever attempted once per open/toggle cycle, regardless of whether the target block exists.

In `_toggleAiMode`, added `_didInitialRestore = false` immediately after setting `_pendingRestorePage`, re-arming the restore for the upcoming content reload.
