## [LOG-004] Tesseract Engine Configuration Update
- **Date**: 2026-04-02
- **Agent**: Antigravity
- **Session type**: Engine Logic Refactoring
- **What was done**:
  - Investigated the usage of `ben+eng` combined layout logic within the application.
  - Modified `LeafPlugin.kt` to completely eliminate the combined `ben+eng` Tesseract initialization.
  - Tesseract is now configured to load the `ben` and `eng` models purely and independently.
- **Why**: Requested by the user because mixing `ben+eng` results in poor Bengali OCR processing. Tesseract should use pure `ben` for Bengali reading without introducing English bias where it's not strictly necessary. 
- **Files changed**:
  - `android/app/src/main/kotlin/com/spark/leaf/LeafPlugin.kt`
- **Outcome**: The OCR engine now natively spins up clean, separate instances for `ben` and `eng` directly during initialization. Bengali documents will hit the pure `ben` engine.

## [LOG-004b] Page Persistence Restore — Bug Fix
- **Date**: 2026-04-02
- **Agent**: Antigravity
- **Session type**: Bug Fix
- **Bug**: BUG-005
- **What was done**:
  - Traced the page persistence failure end-to-end: DB schema has `last_read_page`, DAO `updateLastReadPage` writes it, `bookProvider` re-fetches it after invalidation — all correct.
  - Identified two bugs in `_ReaderScreenState._restoreToPage` in `reader_screen.dart`:
    1. `_didInitialRestore` was only set **after** `Scrollable.ensureVisible`, so if the saved page wasn't found in any block (`index == -1`), the flag was never set. The 2s `readerRefreshProvider` timer caused `_scheduleRestore` to re-queue endlessly every rebuild.
    2. `_toggleAiMode` set `_pendingRestorePage` to re-scroll after content reload, but the fix to Bug A made `_didInitialRestore` permanently `true`, silently skipping that restore.
- **Fix**:
  - Moved `_didInitialRestore = true` and `_pendingRestorePage = null` to the **top** of `_restoreToPage`, before any early-return checks.
  - Added `_didInitialRestore = false` in `_toggleAiMode` after setting `_pendingRestorePage`, to re-arm the restore for AI mode content reloads.
- **Files changed**:
  - `lib/ui/screens/reader/reader_screen.dart`
- **Outcome**: Restore now fires exactly once on open (and once per AI mode toggle). The 2s refresh timer no longer causes scroll position fighting.

## [LOG-004c] Page Persistence — Second Pass Fix (Context Timing)
- **Date**: 2026-04-02
- **Agent**: Antigravity
- **Session type**: Bug Fix (follow-up to BUG-005)
- **What was done**:
  - Identified a second, deeper timing bug: even with the eager flag fix from LOG-004b, the restore still silently no-ops. Reason: `_scheduleRestore` was called from inside `build()`. The post-frame callback fired during the *very first frame* after the ListView renders, at which point `_blockKeys[index].currentContext` is `null` (widgets created but not yet laid out). Since `_didInitialRestore` was now set eagerly, this null-context early-return permanently locked out any future restores.
  - Eliminated `_scheduleRestore` from `build()` entirely.
  - Added `didChangeDependencies` override with a `ref.listen` on `readerViewProvider`, which fires only on the `loading → data` transition — the correct Riverpod pattern for side effects.
  - Added `_tryScrollToPage` with a boolean return: `false` means the context wasn't ready yet. Added a 250ms retry via `_restoreRetryTimer` to handle the first-frame layout race.
  - Added `_restoreRetryTimer` cancellation in `dispose()`.
- **Files changed**:
  - `lib/ui/screens/reader/reader_screen.dart`
- **Outcome**: Restore is now triggered precisely once per session open via `ref.listen`, with a guaranteed retry if the ListView hasn't laid out yet. No more dependency on side effects from within `build()`.
